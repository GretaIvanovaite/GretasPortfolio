Animated Easter Card Creator

A multi-step web application for creating and sending personalised animated Easter cards in bulk.

  • HTML5
  • CSS3
  • JavaScript
  • CSS Animations
  • CSV / JSON

Overview

The Easter Card Creator is a browser-based tool that lets users build, personalise and send animated Easter cards to multiple recipients in a single workflow. The project began as an exploration of CSS animation combined with a practical sending mechanism, and grew into a multi-step application with file upload, live preview and bulk dispatch via email.

Like following a crochet pattern, the application guides users through a fixed sequence of steps: choose a design, upload a recipient list, preview the result and send. Each step builds on the last, and skipping one would unravel the whole thing.

The Problem

Sending personalised seasonal cards at scale is time-consuming. Most tools require a paid subscription, a design account or are locked to specific platforms. The goal was to build a self-contained, browser-only solution that any user could run locally without accounts or services.

The Solution

A three-stage wizard: design selection, recipient upload and preview, then send. Recipients are loaded from a CSV or JSON file. The application reads each row, injects the recipient name into the card template and generates a personalised mailto link for dispatch. Animations are pure CSS with JavaScript controlling timing and sequencing.

Development Process

The animation layer was built first as a standalone proof of concept, using CSS keyframes for the hatching egg sequence and a bouncing chick. Once the animation was stable, the multi-step form logic was layered on top: step tracking, validation between stages and a reset mechanism.

The file parsing component uses the FileReader API to read CSV and JSON uploads client-side, avoiding any server requirement. Error handling covers missing columns, empty files and malformed data.

Outcome and Reflection

The finished application handles the full workflow in a single HTML file with no dependencies. The main learning was around CSS animation sequencing: keyframe timing is not forgiving, and a single percentage off in a multi-stage animation reads as a bug. That is the dropped stitch problem in a different medium.

If rebuilt, the animation layer would be driven by JavaScript timing rather than CSS alone, giving finer control over sequencing and making it easier to sync with user interactions.
