Animated Easter Card Creator

A multi-page tool for businesses to create and send personalised animated Easter cards in bulk, built as a single HTML file with no external dependencies.

  • HTML5
  • CSS3
  • JavaScript
  • CSS Animations
  • CSV / JSON

Overview

A browser-based tool for businesses to send personalised seasonal cards to a recipient list without needing a subscription service or a developer. The project is split across two files: the creator tool the company uses to configure and dispatch cards, and a separate recipient-facing card page that receives personalised content via URL parameters. The company works through five steps; each recipient receives an email link that opens directly to their personalised animated card.

The Problem

The brief specified several hard constraints that ruled out conventional multi-page site structures and shaped every technical decision from the start.

  • Personalised seasonal card tools require paid subscriptions, design accounts or specific platforms, making them inaccessible to small businesses without technical staff
  • Brief requirement: fully self-contained and browser-only, with no accounts, no installs and no network connection required at runtime
  • Hard technical constraint: the entire creator had to be a single HTML file with no separate stylesheets and no external scripts

Requirements Gathering

Requirements were derived directly from the brief specification and the constraints it imposed. The brief stipulated a single self-contained HTML file with no external dependencies, which immediately ruled out frameworks, separate stylesheets and any server-side logic. The personalisation requirement raised a further question: how do you pass recipient-specific data to a static card file with no server? This led to the URL parameter architecture, which became a functional requirement rather than an implementation detail. The forward-only flow requirement emerged from analysing the failure modes of multi-step forms: allowing users to jump between steps or navigate backward before completing each stage introduces invalid state that is difficult to recover from cleanly.

Functional

  • Accept a CSV or JSON file upload from the user's own device
  • Parse the uploaded file and extract column headers for user selection
  • User selects which column contains email addresses; the column is validated to confirm it contains email-format values
  • User optionally selects a name column for personalisation
  • User selects a phrase from preset templates containing {company} and {name} placeholders, substituted per recipient at send time
  • Preview a sample personalised card before any dispatch takes place
  • Generate one personalised mailto link per recipient with the card URL and recipient data encoded as parameters
  • The recipient card renders the personalised animation from URL parameters alone, independently of the creator tool

Non-functional

  • The creator tool must be a single self-contained HTML file with no separate stylesheets or external scripts (brief hard constraint)
  • No server or backend required at any point in the flow
  • Forward-only step navigation: users cannot skip ahead or navigate backward before the preview step
  • The recipient card must be independently accessible by URL without the creator tool

Design

Two-file architecture

  • index.html: the creator tool the company uses
  • card.html: the recipient-facing card, receives URL parameters

Keeping them separate meant the animation could be built and polished independently from the sending logic, and kept the recipient file clean and fast to load.

Creator flow

A strict forward-only sequence was implemented up to the preview step in order to prevent invalid state from accumulating across steps, rather than attempting to validate or correct it reactively at dispatch time.

  1. Landing page
  2. Company details input (name, website URL, CSV/JSON upload)
  3. Card customisation (email column, name column, phrase selection)
  4. Preview
  5. Sending

Animation sequence

Planned on paper before any CSS was written, each beat mapped to a keyframe percentage before touching code.

  1. Egg rolls in from a distance toward the viewer
  2. Egg cracks apart with a burst of confetti
  3. Easter bunny climbs out
  4. Bunny retrieves items from inside the broken egg: basket, flowers, smaller eggs
  5. Bunny pulls out a scroll; scroll unravels and zooms toward the viewer showing the personalised message and optional company link

Other design decisions

  • Spring palette (yellows, soft greens, warm pinks), chosen to match the Easter seasonal context without the card feeling generic
  • System sans-serif font stack selected in order to avoid the external font dependency that would violate the single-file constraint on card.html
  • Step progress indicator implemented via CSS counter on an ordered list in order to avoid JavaScript class toggling, keeping the navigation logic minimal
  • JSON support added alongside CSV using the same FileReader call with a different parse path, in order to cover CRM exports as well as standard spreadsheet outputs

The Solution

File parsing (input step)

  • FileReader API reads the file entirely client-side, data never leaves the browser
  • UTF-8 BOM characters stripped before parsing, prevents Excel-exported files from producing a corrupt first column header
  • Empty rows filtered out, prevents blank recipient entries

Customisation (selections step)

  • Email column: company chooses which column contains addresses; application validates the column actually contains valid email format, preventing silent misconfiguration that would produce undeliverable emails
  • Name column: optional; enables the card to address each recipient individually
  • Phrase: chosen from preset templates with {company} and {name} placeholders, substituted at send time per recipient

Data sanitisation before dispatch

  • String.trim() on names and phrase values, removing whitespace
  • encodeURIComponent() on all URL parameters; without encoding, a name containing &, = or # would corrupt the URL structure and pass the wrong value to card.html

Per-recipient personalisation

Each recipient gets one mailto link. The email body contains a link to card.html with their name, the substituted phrase and the company's details as URL parameters. card.html reads those parameters on load and injects them into the animation, no server, no database.

Development Process

  • Animation built first: standalone proof of concept on card.html, isolated from creator logic; once the full sequence was stable and timing felt right, the creator was built on top
  • Scroll reveal timing: most complex challenge, the text must be readable before the zoom keyframe begins; small changes to earlier phases shifted every percentage that followed. Solution: treat each beat as a fixed anchor point and calculate percentages backward from the end
  • Wizard focus management: on each step advance, JavaScript sets tabindex="-1" on the incoming step's heading and calls .focus(); without this, screen reader users advance through the form with no indication the page has changed
  • ARIA error handling: aria-invalid="true" + aria-describedby on the file upload input; if the file has no recognisable columns, is structurally invalid or is empty, a descriptive error message names the specific problem
  • Single-file organisation: script block divided into clearly labelled sections: step navigation, file parsing, URL parameter construction, mailto dispatch

Outcome and Reflection

Complete workflow across two files with no external dependencies, company details, customisation, preview and bulk dispatch on the creator side, personalised animated sequence on the card side.

Two things I would revisit:

  • Animation control: CSS @keyframes run on the compositor thread and cannot be paused or reset mid-sequence without cloning and replacing the animated element. A setTimeout/clearTimeout approach would give finer control: pausing at a specific beat, scrubbing, or synchronising the zoom to a user click
  • Dispatch scale: most browsers cap sequential mailto opens before requiring explicit user permission, making the tool practical for roughly 20 recipients. Replacing mailto with an email API such as EmailJS would remove that ceiling and allow a formatted HTML email rather than plain text
