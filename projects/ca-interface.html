Collections & Archives (C&A)

Final year research and development project: a "generous" interface for exploring over 2 million objects using a museum REST API, featuring an AI discovery assistant. Deployed on Netlify.

  • HTML5
  • CSS3 (Flexbox & Grid)
  • JavaScript (ES6+)
  • REST API
  • AI (Qwen 2.5)
  • Netlify

Overview

Collections & Archives (C&A) is a final year project exploring how a "generous" interface design can help people discover museum objects more easily. Instead of starting with a blank search box, the prototype shows a large grid of items from a museum collection immediately. It also includes an AI assistant to help answer questions about the objects.

A "generous interface" is a design that shows a lot of content upfront to support browsing and accidental discovery. It is the opposite of a simple search box, which requires you to already know what you are looking for.

The Problem

The project was inspired by a gap between the rich data available in a museum's open API and how the official website presents it to casual visitors.

  • The museum API has over 2 million objects, but the website starts with a search box that can be intimidating if you do not have a specific item in mind.
  • Research shows that casual visitors often prefer browsing through images rather than typing search terms.
  • A small study confirmed that people often feel "stuck" when they arrive at the museum site without a clear plan.

Requirements Gathering

Requirements were gathered through four stages: a literature review, a study of 20 other digital museum platforms, a questionnaire with 23 people, and a test of the official museum website. The questionnaire showed that people enjoy visual exploration and are interested in AI features like contextual explanations. The user study found that people scrolled 300% more when they were free to explore compared to when they were doing a specific search. These results led to a set of core functional and technical requirements.

Core Features

  • Visual Homepage: Show museum objects immediately on arrival without requiring a search.
  • Rich Item Pages: Detailed views with metadata and related object suggestions.
  • Navigation Pathways: Multiple ways to enter the collection, such as by Origin or Category.
  • AI Chat Assistant: A helper that can answer questions about the object you are currently viewing.
  • Live API Integration: Real-time connection to the museum REST API.

Technical Goals

  • Performance: An optimized strategy to keep page loads fast while staying within API limits.
  • Clear AI Labeling: Ensuring users know they are talking to an AI model grounded in real museum data.
  • Privacy: No user data is stored or tracked beyond the current session.

Design

The design is "Desktop-First" because showing a large amount of visual content is easier on a big screen. The prototype focuses on the 1280px experience while still keeping a basic, working view for mobile users. Initial sketches and wireframes helped map out a dense grid layout that supports effortless exploration.

Browsing Pathways

Based on an analysis of the API data, four main ways to browse were chosen: Collections, Categories, Materials, and Origins. Other options like "Date" or "Creator" were left out because the data was too inconsistent to provide a good browsing experience.

The Solution

API Strategy

The app uses a two-step approach: first, it fetches a lightweight list of thumbnails for the main grid. It only calls for the full object details when a user clicks to open an item. This keeps the initial loading time fast.

AI Assistant

The chatbot uses the Qwen 2.5 model via the Hugging Face API. It is an open-source model that works well for answering specific questions about museum objects. A "Netlify function" is used as a bridge to keep the API key safe. The AI is restricted to only talk about museum topics and is told to say when it does not have enough information rather than making things up.

Development Process

  • Figma to Code: High-fidelity designs were created for every page before any code was written.
  • Defensive Data Handling: Real objects were sampled to see which fields (like "Maker") were often missing, so fallback text like "Maker unknown" could be added.
  • User Testing: A plan for comparative testing with 10+ people to see how the prototype performs against the official museum website.

Outcome and Reflection

The project is live at collectionsarchives.netlify.app. It successfully demonstrates how a browse-heavy design can make a massive museum collection feel more accessible to everyone. By using data from a museum REST API, Collections & Archives provides a realistic environment for evaluating how users interact with large-scale archive data when presented visually.

One area for future improvement is the "Load More" system. While it works for adding more items to the grid, focus management could be improved to help keyboard users find the new content more easily. Transitioning to a true infinite scroll pattern could also make the exploration feel even more seamless.
